В чем проблема вашего текущего подхода?
Сейчас у вас есть два места, где "спрятана" одна и та же логика:

В Program.cs (внутри ActionBlock): 
Вы создаете ActionBlock и прямо в нем пишете, что делать с данными: 
"получить сервис UiUpdateService из контейнера и вызвать его метод".

В UiUpdateBackgroundWorker: 
Вы внедряете этот же UiUpdateService в конструктор, хотя ни разу его не используете. Класс просто ждет завершения блока.
Это создает избыточность и запутывает. 
Если вы захотите изменить логику обработки данных (например, добавить кэширование или логирование перед вызовом UiUpdateService),
вам придется менять код в двух разных местах. Это нарушает принцип DRY (Don't Repeat Yourself).

В чем суть предложенного исправления?
Предложенный вариант предлагает разделить обязанности еще более четко:

ActionBlock (в Program.cs) — это "исполнитель". 
Его единственная работа — взять объект и передать его нужному сервису. Он знает о бизнес-логике (UiUpdateService).

UiUpdateBackgroundWorker — это "сторож" или "менеджер". 
Его единственная работа — следить за тем, чтобы приложение не закрылось, 
пока его блок (_uiUpdateBlock) не закончит обработку всех данных.
Он не должен ничего знать о бизнес-логике.
По сути, BackgroundWorker — это просто "обертка" для блока данных, которая позволяет ему работать в инфраструктуре .NET Host.

Что конкретно нужно сделать?
Вам нужно заменить ваш текущий класс UiUpdateBackgroundWorker.cs на предложенный новый вариант.

Шаг 1: Замените содержимое файла UiUpdateBackgroundWorker.cs

Удалите старый код и вставьте этот:

csharp
Копировать
using Microsoft.Extensions.Hosting;
using Microsoft.Extensions.Logging;
using System.Threading;
using System.Threading.Tasks;
using System.Threading.Tasks.Dataflow; // Убедитесь, что этот using есть

namespace TradingTerminal.Worker.Services
{
    public class UiUpdateBackgroundWorker : BackgroundService
    {
        private readonly ILogger<UiUpdateBackgroundWorker> _logger;
        private readonly ActionBlock<object> _uiUpdateBlock;

        // Внедряем ТОЛЬКО логгер и сам блок данных.
        // Зависимость от UiUpdateService УДАЛЕНА, так как она уже есть в ActionBlock.
        public UiUpdateBackgroundWorker(
            ILogger<UiUpdateBackgroundWorker> logger,
            ActionBlock<object> uiUpdateBlock)
        {
            _logger = logger;
            _uiUpdateBlock = uiUpdateBlock;
        }

        protected override async Task ExecuteAsync(CancellationToken stoppingToken)
        {
            _logger.LogInformation("Сервис обновления UI запущен и ожидает данные.");
            
            try
            {
                // Ждем, пока источник данных не вызовет Complete()
                // и все сообщения не будут обработаны в делегате ActionBlock
                await _uiUpdateBlock.Completion;
            }
            catch (Exception ex)
            {
                // Хорошая практика - логировать ошибки в блоках Dataflow
                _logger.LogError(ex, "Произошла ошибка в блоке обновления UI.");
            }
            
            _logger.LogInformation("Сервис обновления UI корректно остановлен.");
        }
    }
}

Шаг 3: Проверьте регистрацию в Program.cs

Убедитесь, что вы регистрируете воркер простым способом. 
Контейнер DI сам подставит ActionBlock, который вы уже зарегистрировали как Singleton.

csharp
Копировать
// Эта строка уже должна быть у вас в Program.cs
builder.Services.AddSingleton(uiUpdateBlock); // Регистрация самого блока

// Регистрация воркера остается простой:
builder.Services.AddHostedService<UiUpdateBackgroundWorker>();
Итог
После этих изменений ваш код станет чище. 
Класс UiUpdateBackgroundWorker будет делать только то, что должен — управлять жизненным циклом. 
Вся бизнес-логика останется там, где ей и место — внутри делегата ActionBlock. 
Это делает систему более понятной и легкой для поддержки.
