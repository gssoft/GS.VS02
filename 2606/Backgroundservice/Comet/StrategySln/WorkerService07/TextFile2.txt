Как работает этот код (Разбор архитектуры)
1. Отсутствие интерфейсов
В блоке ConfigureServices нет ни одного слова interface. 
Мы регистрируем «голые» типы Func<> и Action<>. Это максимально легковесный подход.

2. Магия композиции в конструкторе MyDelegatingWorker
Когда хост создает new MyDelegatingWorker(...), контейнер внедрения зависимостей (DI) сам подставляет нужные лямбды. 
Giga
https://giga.chat/link/gcsYtzLwsk

Конструктор получает не объекты-классов, а готовые к вызову функции. 

3. Инверсия управления в _scheduler
Обратите внимание на сигнатуру _scheduler:
Func<TimeSpan, CancellationToken, Func<CancellationToken, Task>, Task>
Это стратегия высшего порядка. Сам планировщик не знает, что делать. 
Ему передают функцию workAction («вот тебе кусок кода, запусти его согласно моему расписанию»). 
Внутри ExecuteAsync мы передаем ему анонимную локальную функцию (лямбду), которая содержит вызов _behavior и _metrics.

4. Иммутабельность в TunerConfig
Использование record для конфигурации позволяет писать безопасный код:
return config with { Gain = config.Gain * 0.95 };
Мы не меняем исходный объект initialConfig, а создаем новую копию с обновленным значением. 
Это стандарт современной разработки на C#.

Преимущества этого подхода над интерфейсами
Критерий	Интерфейсы (Классы)	Делегаты (Лямбды)
Количество файлов	Много (по файлу на каждый I...Strategy)	Один файл (Program.cs)
Нагрузка на GC	Выше (создаются экземпляры классов)	Ниже (делегаты легче)
Инкапсуляция полей	Можно хранить приватное состояние внутри класса	Нужно передавать через замыкание или параметры
Читаемость	Легко искать реализации по названию интерфейса	Код находится прямо в точке регистрации (DI)
Этот вариант идеален для микросервисов и фоновых задач, где поведение алгоритмов простое и не требует сложной иерархии состояний.
