Проблема заключается в том, что классы PeriodicProcessor, ContinuousListener, StartupInitializer и CronJob,

Анализ ошибки
Ошибка CS0246 «The type or namespace name 'X' could not be found» означает, 
что компилятор ищет класс/тип по имени X в подключенных пространствах имён и сборках, но не находит его.

В файле Program.cs вашего консольного приложения вы регистрируете сервисы:

Компилятор смотрит на директиву using DS.BackgroundServices.Core02;. 
Это значит, он ожидает найти эти классы внутри проекта-библиотеки DS.BackgroundServices.Core02.
Однако при анализе файлов этой библиотеки видно, что там определены другие классы:

Суть архитектуры

Они не делают полезную работу сами. Их задача — управлять жизненным циклом сервиса:

Как правильно стартовать (логирование, вызов хуков).
Как организовать основной цикл работы (бесконечный цикл, цикл с задержкой, ожидание по CRON-расписанию).
Как корректно обрабатывать ошибки (не падать от одной ошибки итерации).
Как безопасно останавливаться (асинхронная очистка ресурсов).
Вся эта сложная «обвязка» уже написана за вас. Вам не нужно беспокоиться о том, 
как правильно остановить сервис при закрытии приложения или как залогировать необработанное исключение.

Ваши выводы абсолютно точны и отражают суть этой архитектурной парадигмы. 
Давайте закрепим их, чтобы «вопросительный знак» превратился в восклицательный.

1. DI не регистрирует абстрактные классы
Это фундаментальное правило работы большинства контейнеров внедрения зависимостей (DI), включая Microsoft.Extensions.DependencyInjection.

Причина: Абстрактный класс — это «недоделка», он не может быть создан с помощью оператора new. 
Нельзя написать var service = new BackgroundServiceBase(...), потому что компилятор этого не позволит.
Что делает DI: Когда вы пишете builder.Services.AddSingleton<IMyService, MyService>(), 
вы даёте контейнеру инструкцию: «Если кто-то попросит у тебя IMyService, создай для него экземпляр класса MyService». 
Для создания экземпляра нужен конкретный (конкретизированный) тип.
Как это работает здесь: Вы никогда не регистрируете PeriodicBackgroundService. Вы регистрируете его наследника:
csharp
Копировать
// В Program.cs вашего консольного приложения
builder.Services.AddHostedService<CustomDataProcessor>(); 
  А уже внутри самого CustomDataProcessor заложена логика от базового класса. 
Контейнер создаёт CustomDataProcessor, а при выполнении его методов вызывается код из PeriodicBackgroundService.
2. Нормальность абстрактных классов в библиотеках
Для многих разработчиков первая реакция на такую библиотеку — недоумение: «Почему мне продали недостроенный дом? 
Где готовые комнаты?»
Но в мире больших систем это стандарт де-факто. Это называется созданием Framework'а или Kernel-библиотеки.

Аналогия из реального мира
Представьте компанию по производству станков.

Подход «готовой библиотеки»: 
Компания привозит вам готовый станок для вытачивания гаек М6. Он идеален, если вам нужны гайки М6. 
Но если завтра понадобятся болты М8 — этот станок бесполезен.
Подход «абстрактных классов» (наш случай): Компания привозит вам ЧПУ-станину с контроллером, системой подачи СОЖ и идеально откалиброванными осями. 
Станина сама по себе ничего не производит. Но она предоставляет API: «Вот сюда крепится шпиндель, вот так подаётся заготовка, вот команда для начала движения».
Теперь ваша задача — спроектировать и изготовить только насадку (шпиндель), которая будет делать вашу уникальную деталь. 
Сама станина решает все общие проблемы: стабильность, охлаждение, точность координат.
В нашем случае BackgroundServiceBase — это та самая ЧПУ-станина. 
Она решает сложнейшие проблемы асинхронного старта/остановки, перехвата исключений и логирования жизненного цикла. 
Вам остаётся лишь «изготовить шпиндель» — метод DoWorkAsync.

Поэтому ваш вывод «Век живи — век учись» — самый главный. То, что кажется странным в маленьких учебных проектах, 
оказывается мощнейшим инструментом масштабирования и поддержки в энтерпрайз-разработке.

Теперь наш класс принимает два дженерика. Обратите внимание на ограничение where TTunes : class, new().

csharp

public class Processor<TBehavior, TTunes> : BackgroundService 
    where TBehavior : IWorkBehavior<TTunes>, new()
    where TTunes : class, new()
{
    private readonly ILogger _logger;
    private readonly IOptions<TTunes> _tunesOptions; // Получаем настройки через DI
    private readonly TBehavior _behavior;

    public Processor(
        ILogger<Processor<TBehavior, TTunes>> logger,
        IOptions<TTunes> tunesOptions)
    {
        _logger = logger;
        _tunesOptions = tunesOptions;
        _behavior = new TBehavior(); // Создаём поведение
    }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        var settings = _tunesOptions.Value;
        
        _logger.LogInformation("Запуск процессора с настройками: {@Settings}", settings);

        // Передаём настройки внутрь поведения!
        await _behavior.DoWorkAsync(stoppingToken, settings);
    }
}

Ответ

Да, это очень зрелое и правильное архитектурное решение. 
Разделение на Workers (конкретные реализации) и WorkersAbstract (базовые классы/фреймворк) — 
это классический паттерн для создания библиотек многократного использования (reusable libraries).

Давайте проанализируем эту структуру с точки зрения лучших практик.

Что было достигнуто этим разделением?
Чёткое разделение ответственности

Папка WorkersAbstract: Это «движок», или ядро библиотеки. Здесь находится сложная логика управления жизненным циклом (ExecuteAsync, 
обработка ошибок, асинхронная остановка). Эти классы помечены как abstract, потому что они не предназначены для самостоятельного использования. 
Их задача — предоставить каркас (framework) для других классов.
Папка Workers: Это примеры конкретных реализаций, которые показывают, как пользоваться движком из первой папки. 
Они наследуются от абстрактных классов и реализуют единственный метод DoWorkAsync, где и содержится уникальная бизнес-логика.
Создание публичного API библиотеки
Разработчик этой библиотеки говорит пользователю:

«Не трогайте ничего в WorkersAbstract. Считайте, что этого кода нет. 
Ваша точка входа — это конкретные классы в папке Workers (PeriodicProcessor, CronJob и т. д.). 
Если их поведение вас не устраивает, вы можете создать свой собственный класс рядом с ними, унаследовавшись от любого класса из WorkersAbstract».

Инкапсуляция сложности
Пользователь вашего консольного приложения (CA_Background_02) теперь видит только простые конечные классы.
Ему не нужно разбираться, как работает цикл внутри ScheduledBackgroundService или как парсится CRON-выражение. 
Он просто регистрирует CronJob и получает работающий сервис.

Соответствие вашему предыдущему предложению
Эта структура является эволюцией вашей идеи про <TBehavior>.
Вместо того чтобы делать один универсальный класс Processor<TBehavior>,

Пользователю всё ещё нужно выбрать правильный базовый класс (ваше "поведение"), но ему не нужно создавать отдельный класс-обёртку для настроек.

Консистентность	Несоответствие комментария и кода в BackgroundServiceBase.	Исправить порядок вызовов или текст комментария.
Настройки	Параметры (интервалы, cron-строки) захардкожены в конструкторах примеров.	
Переписать конструкторы потребителей на приём IOptions<T> или создать статические фабричные методы Create().
DI Registration	Регистрация сервисов идёт поштучно.	Создать extension-метод для IServiceCollection, 
чтобы регистрация выглядела элегантнее: services.AddMyBackgroundServices();.
Nullability	Используется <Nullable>enable</Nullable>, но нет аннотаций [NotNull] / [MaybeNull] на параметрах ILogger.	
Усилить null-safety analysis, особенно на границах API. 
Например: 
protected BackgroundServiceBase([NotNull] ILogger logger) => _logger = logger ?? throw new ArgumentNullException(nameof(logger)); 
(хотя у вас уже есть выброс exception, явные атрибуты помогают статическому анализатору).
Вердикт: архитектурно проект очень силён. Код чистый и идиоматичный для C#/.NET. 
После доработки документации и настройки внешних параметров библиотека будет готова к использованию в продакшене.
