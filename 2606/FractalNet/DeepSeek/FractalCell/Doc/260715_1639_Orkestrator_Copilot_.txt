1 Архитектурная идея и преимущества
Идея

Превратить текущий Orchestrator (фоновой Worker) в Composite Orchestrator, который не только генерирует события, но и агрегирует, управляет и координирует множество IFractalCell как единую логическую единицу.

Преимущества

Единая точка управления: lifecycle, конфигурация, health checks, масштабирование.

Единый контракт для деплоя: Orchestrator как Worker → готов к .NET Aspire / Kubernetes + Dapr.

Гибкая композиция: можно собирать приложения из ячеек, переиспользовать поведения, создавать шаблоны (stateless pool, stateful cluster, leader‑follower).

Интеграция с Dapr: Orchestrator выступает sidecar‑aware компонентом, адаптирующим pub/sub, state, bindings.

2 Дизайн и ключевые интерфейсы
Основные сущности

CompositeOrchestrator — главный класс, реализует IHostedService / IFractalCell или новый ICompositeCell.

ICompositeCell — API для управления группой ячеек.

CompositionPolicy — стратегия маршрутизации/распределения событий внутри композиции.

BehaviorScope — область действия поведения: Local, Composite, Global.

3 Интеграция с Worker Template и .NET Aspire / Dapr
Почему Worker → .NET Aspire удобно

Worker уже реализует IHostedService и фоновые циклы — идеально для контейнерного запуска.

.NET Aspire (или аналог) ожидает deployable units с lifecycle hooks — Orchestrator как Worker подходит «из коробки».

Dapr / sidecar

Orchestrator реализует адаптеры:

Pub/Sub: подписка на топики Dapr, публикация агрегированных событий.

State: хранение глобального состояния композиции в Dapr state store.

Service Invocation: вызовы внешних сервисов и экспонирование endpoint’ов.

Корреляция: Orchestrator обязан прокидывать CorrelationId и trace context в события.

Практический паттерн

Запуск: Orchestrator стартует, регистрирует себя в Dapr (если нужно), создаёт/подключает члены, подписывается на топики.

Взаимодействие: внешние события приходят в Orchestrator → он применяет CompositionPolicy → распределяет события по членам или обрабатывает на уровне композиции.

4 Основные трудности и способы их решения
1. Согласованность состояния

Проблема: члены композиции имеют локальное состояние.

Решение: явная модель владения (owner), использовать Dapr state store для критичных данных, leader election для координации.

2. Жизненный цикл и отказоустойчивость

Проблема: частичные сбои членов.

Решение: health checks, graceful shutdown, автоматическое восстановление, circuit breaker для внешних вызовов.

3. Маршрутизация и производительность

Проблема: горячие точки при broadcast.

Решение: конфигурируемые политики (sharding, rate limiting), backpressure, batching.

4. Наблюдаемость и трассировка

Проблема: трассировка через уровни.

Решение: OpenTelemetry, единый CorrelationId, централизованные метрики и логи.

5. Версионирование контрактов

Проблема: изменение схем событий ломает интеграции.

Решение: EventType + EventVersion, трансформеры в Orchestrator, backward compatibility.

6. Безопасность

Проблема: доступ к секретам и вызовам.

Решение: secrets через Dapr; авторизация/аутентификация на уровне Orchestrator.

6 Минимальный пример кода CompositeOrchestrator
csharp
public class CompositeOrchestrator : ICompositeCell
{
    public string CompositeId { get; }
    private readonly List<IFractalCell> _members = new();
    private readonly ICompositionPolicy _policy;
    private readonly ILogger _logger;

    public CompositeOrchestrator(string id, ICompositionPolicy policy, ILogger logger)
    {
        CompositeId = id;
        _policy = policy;
        _logger = logger;
    }

    public IReadOnlyList<IFractalCell> Members => _members.AsReadOnly();

    public async Task AddMemberAsync(IFractalCell cell, CancellationToken ct = default)
    {
        _members.Add(cell);
        await cell.InitializeAsync();
        await cell.StartAsync(ct);
        _logger.LogInformation("Member {Id} added to composite {Composite}", cell.CellId, CompositeId);
    }

    public async Task RemoveMemberAsync(string cellId, CancellationToken ct = default)
    {
        var cell = _members.FirstOrDefault(c => c.CellId == cellId);
        if (cell != null)
        {
            await cell.StopAsync(ct);
            _members.Remove(cell);
        }
    }

    public async Task BroadcastAsync(IApplicationEvent @event)
    {
        var targets = await _policy.SelectTargetsAsync(@event, Members);
        foreach (var t in targets) await t.ExternalBus.SendToCellAsync(t.CellId, @event);
    }

    // IHostedService
    public Task StartAsync(CancellationToken cancellationToken) => Task.CompletedTask;
    public Task StopAsync(CancellationToken cancellationToken) => Task.CompletedTask;
}
Это минимальный скелет — в реальности нужно обработать ошибки, токены отмены, health, конфигурацию и Dapr‑адаптеры.
