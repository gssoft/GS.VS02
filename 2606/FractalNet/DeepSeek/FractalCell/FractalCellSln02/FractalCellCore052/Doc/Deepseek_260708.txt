1. Текущая реализация ExternalBus (что есть сейчас)
В версиях 04 и 05 каждая FractalCell имеет свой собственный экземпляр ExternalBus (например, ChannelExternalBus или TplExternalBus).
Но все эти ExternalBus'ы подключаются к одному общему IFractalEventHub (в памяти – InMemoryFractalEventHub), который зарегистрирован как синглтон в DI-контейнере.

text
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│  Cell A     │      │  Cell B     │      │  Cell C     │
│ ExternalBus │      │ ExternalBus │      │ ExternalBus │
└──────┬──────┘      └──────┬──────┘      └──────┬──────┘
       │                    │                    │
       └──────────┬─────────┴──────────┬─────────┘
                  │                    │
            ┌─────▼─────┐        ┌─────▼─────┐
            │  Hub (общий)       │ (общий)   │
            └───────────┘        └───────────┘
Таким образом, фактически все ячейки общаются через один общий канал – Hub. Каждый ExternalBus – это лишь прокси-обёртка над Hub'ом, которая предоставляет методы SendToCellAsync и ReadAllAsync для конкретной ячейки.

Вывод: уже сейчас у вас есть единый роутер (Hub), и ячейки не имеют отдельных физических шин – они лишь используют общий ресурс.

Сервис (FractalCell) — это автономная единица, содержащая:

Набор поведений, реализующих бизнес-логику.

Внешняя шина (ExternalBus / Hub) — это канал связи, который может быть:

Общим для группы сервисов (одноранговых).

Специализированным для конкретной задачи (например, шина для обработки заказов, шина для логирования).

Иерархическим (если нужно) — но это лишь частный случай графа.

Граф связей:

Сервисы подключаются к нужным шинам.

Шины могут быть соединены между собой (мосты) для передачи сообщений между разными кластерами.

Это даёт максимальную гибкость: можно строить любые топологии — звезда, кольцо, сетка, иерархия.

Преимущества:

Минимальное дублирование логики — каждый сервис делает одну вещь и делает её хорошо.

Лёгкость сборки приложений — выбираем нужные сервисы и шины, соединяем их.

Простота масштабирования — добавляем новые сервисы или шины без изменения существующих.

Возможность построения pipeline'ов — цепочек обработки через последовательные шины.

Для этого создадим новое поведение, например OrchestratorBehavior, которое будет подписываться на события, принимать решения и отправлять команды через внешнюю шину.

В текущей реализации Worker выполняет оркестрацию, отправляя случайные события. Мы можем перенести эту логику в поведение, прикрепить его к какой-то ячейке (например, Root), и тогда Worker станет не нужен для оркестрации (он будет только запускать систему).

Мы можем добавить третье поведение к Root — OrchestratorBehavior, или создать отдельную ячейку Orchestrator с этим поведением. Второй вариант более чистый — отдельная ячейка, которая отвечает только за оркестрацию.

В OrchestratorBehavior реализовать логику: например, каждые 3 секунды отправлять случайное событие (Heartbeat или ProcessData) на случайную ячейку. Это будет дублировать текущую логику Worker'а.

Также нужно убедиться, что ячейки могут общаться через общий Hub (уже есть).

Мы можем не удалять старый Worker-оркестратор, а просто закомментировать его вызов или сделать опциональным. Для демонстрации оставим Worker только как инициализатор системы, без цикла отправки событий.

csharp
using FractalCellCore.Core.Interfaces;
using FractalCellCore.Core.Templates;
using Microsoft.Extensions.Logging;

namespace FractalCellApp.Behaviors;

public class OrchestratorBehavior : BackgroundBehaviorTemplate
{
    private readonly Random _random = new();
    private readonly TimeSpan _interval;
    private int _counter;

    public override string BehaviorId => "OrchestratorBehavior";
    public override int Priority => 5; // высокий приоритет

    public OrchestratorBehavior(ILogger<OrchestratorBehavior>? logger = null, TimeSpan? interval = null)
        : base(logger)
    {
        _interval = interval ?? TimeSpan.FromSeconds(3);
    }

    public OrchestratorBehavior() : this(null, null) { }

    protected override async Task BackgroundLoopAsync(CancellationToken ct)
    {
        _logger?.LogInformation("OrchestratorBehavior started, interval: {Interval}", _interval);

        while (!ct.IsCancellationRequested)
        {
            try
            {
                await Task.Delay(_interval, ct);

                if (_attachedCell == null)
                {
                    _logger?.LogWarning("OrchestratorBehavior: no attached cell, skipping");
                    continue;
                }

                // Получаем список активных ячеек из Hub (доступ через ExternalBus? Нужно получить Hub)
                // В текущей архитектуре у ячейки есть ExternalBus, который подключен к Hub.
                // Мы можем получить активные ячейки через ExternalBus, но интерфейс IExternalBus не предоставляет GetActiveCells.
                // Вместо этого мы можем использовать IFractalEventHub, но он не доступен напрямую из поведения.
                // Альтернатива: мы можем хранить список ячеек в поведении (передавать через конструктор или конфигурацию).
                // Или использовать Broadcast, чтобы отправлять события всем, а не конкретной ячейке.
                // Для простоты будем отправлять событие на случайную ячейку, но мы не знаем список.
                // Поэтому лучше использовать Broadcast, чтобы событие получили все ячейки, или использовать заранее известный список.
                // Так как у нас есть только корневая ячейка и дочерние, мы можем передать список CellId в поведение через конструктор или конфигурацию.
                // Однако, чтобы не усложнять, мы можем отправлять события на Root, и Root будет пересылать дальше (как в ConfigureHandlers).
                // Или мы можем использовать ExternalBus.BroadcastAsync, чтобы отправить всем.
                // В текущей реализации ExternalBus имеет BroadcastAsync.
                // Поэтому будем использовать BroadcastAsync, чтобы все ячейки получили событие.
                // Но тогда они все будут обрабатывать, что может быть избыточно.
                // Для демонстрации - используем Broadcast.

                var eventTypes = new[] { "Heartbeat", "ProcessData" };
                var eventType = eventTypes[_random.Next(eventTypes.Length)];

                object eventData = eventType == "ProcessData"
                    ? new { Value = _random.Next(100), Priority = _random.Next(1, 5) }
                    : new { Interval = _random.Next(1, 10) };

                var @event = new FractalEvent(
                    $"{eventType.ToLower()}-{DateTime.UtcNow.Ticks}",
                    DateTime.UtcNow,
                    _attachedCell.CellId,
                    string.Empty, // target не указываем, используем Broadcast
                    eventType,
                    new
                    {
                        Timestamp = DateTime.UtcNow,
                        Source = "OrchestratorBehavior",
                        Data = eventData
                    }
                );

                _logger?.LogInformation("🎛️ [Orchestrator] Broadcasting {EventType} from {CellId}",
                    eventType, _attachedCell.CellId);

                await _attachedCell.ExternalBus.BroadcastAsync(@event);
                _counter++;
            }
            catch (OperationCanceledException)
            {
                break;
            }
            catch (Exception ex)
            {
                _logger?.LogError(ex, "OrchestratorBehavior error");
            }
        }

        _logger?.LogInformation("OrchestratorBehavior stopped, total events sent: {Counter}", _counter);
    }
}

csharp
using FractalCellCore.Core.Interfaces;
using FractalCellCore.Core.Templates;
using Microsoft.Extensions.Logging;

namespace FractalCellApp.Behaviors;

public class OrchestratorBehavior : BackgroundBehaviorTemplate
{
    private readonly Random _random = new();
    private readonly TimeSpan _interval;
    private int _counter;

    public override string BehaviorId => "OrchestratorBehavior";
    public override int Priority => 5; // высокий приоритет (опционально)

    public OrchestratorBehavior(ILogger<OrchestratorBehavior>? logger = null, TimeSpan? interval = null)
        : base(logger)
    {
        _interval = interval ?? TimeSpan.FromSeconds(3);
    }

    public OrchestratorBehavior() : this(null, null) { }

    protected override async Task BackgroundLoopAsync(CancellationToken ct)
    {
        _logger?.LogInformation("🎛️ OrchestratorBehavior started, interval: {Interval}", _interval);

        while (!ct.IsCancellationRequested)
        {
            try
            {
                await Task.Delay(_interval, ct);

                if (_attachedCell == null)
                {
                    _logger?.LogWarning("OrchestratorBehavior: no attached cell, skipping");
                    continue;
                }

                // Выбираем случайный тип события
                var eventTypes = new[] { "Heartbeat", "ProcessData" };
                var eventType = eventTypes[_random.Next(eventTypes.Length)];

                object eventData = eventType == "ProcessData"
                    ? new { Value = _random.Next(100), Priority = _random.Next(1, 5) }
                    : new { Interval = _random.Next(1, 10) };

                var @event = new FractalEvent(
                    $"{eventType.ToLower()}-{DateTime.UtcNow.Ticks}",
                    DateTime.UtcNow,
                    _attachedCell.CellId,
                    string.Empty, // Broadcast – не указываем конкретную цель
                    eventType,
                    new
                    {
                        Timestamp = DateTime.UtcNow,
                        Source = "OrchestratorBehavior",
                        Data = eventData
                    }
                );

                _logger?.LogInformation("📡 [Orchestrator] Broadcasting {EventType} from {CellId}",
                    eventType, _attachedCell.CellId);

                // Отправляем всем через внешнюю шину
                await _attachedCell.ExternalBus.BroadcastAsync(@event);
                _counter++;
            }
            catch (OperationCanceledException)
            {
                break;
            }
            catch (Exception ex)
            {
                _logger?.LogError(ex, "OrchestratorBehavior error");
            }
        }

        _logger?.LogInformation("🏁 OrchestratorBehavior stopped, total events sent: {Counter}", _counter);
    }
}

{
    _logger.LogInformation("🏗️ Initializing fractal system with behaviors...");

    // === Корневая ячейка (два поведения) ===
    var rootBehaviors = new IBehavior[]
    {
        CreateHeartbeatBehavior(),
        CreateDataProcessingBehavior()
    };
    var rootCell = await CreateCellWithBehaviorsAsync("Root", 3, ct, rootBehaviors);
    _cells.Add(rootCell);

    // === Дочерние ячейки ===
    var childConfigs = new List<(string id, int workers, Func<IBehavior> behaviorFactory)>
    {
        ("Child-A", 2, CreateHeartbeatBehavior),
        ("Child-B", 2, CreateDataProcessingBehavior),
        ("Child-C", 1, CreateDataProcessingBehavior)
    };

    {
        var behavior = behaviorFactory();
        var child = await CreateCellWithBehaviorAsync(id, workers, ct, behavior);
        _cells.Add(child);

🔍 Текущая архитектура (в двух словах)
Ячейки (FractalCell) — это автономные узлы, каждый со своей:

Внутренней шиной (InternalBus) — для коммуникации внутри ячейки (воркеры, поведения).

Внешней шиной (ExternalBus) — для взаимодействия с другими ячейками через общий Hub.

Поведения (IBehavior) — подключаются к ячейке и:

Могут обрабатывать события (EventBehaviorTemplate<T>).

Иметь жизненный цикл (ILifecycleBehavior).

Вести фоновую работу (BackgroundBehaviorTemplate).

Оркестратор — это тоже ячейка с поведением (OrchestratorBehavior), которая генерирует события и рассылает их через BroadcastAsync. Это пример того, как ячейка может быть не только потребителем, но и источником управления.

Worker — теперь только инициализатор системы. Он создаёт ячейки, стартует их и затем ожидает завершения. Вся оркестрация вынесена в ячейки.

Hub (InMemoryFractalEventHub) — центральный маршрутизатор, который обеспечивает доставку событий между ячейками (по идентификатору ячейки или широковещательно).

Во время работы ячейки можно добавлять или удалять поведения (через AddBehavior / RemoveBehavior), что даёт горячую замену логики.

В FractalCellFactory.cs есть методы CreateWithBehaviorAsync и CreateWithBehaviorsAsync. В комментариях есть путаница: "Поведения уже добавлены в ячейку через конструктор, поэтому AttachToAsync не нужен." — но затем они всё равно вызывают behavior.AttachToAsync(cell, ct) для каждого поведения. Это дублирование: сначала поведения добавляются через конструктор (вызов AddBehavior), затем AttachToAsync. Это нормально, но стоит отметить, что AttachToAsync также может вызывать подписку на внутреннюю шину (если мы это реализуем). В текущей версии AttachToAsync только устанавливает ссылку и вызывает OnAttachedAsync — это безопасно.

В FractalCellTemplate.cs — роутер работает, жизненный цикл есть. В ExecuteAsync он читает только из внешней шины. Это значит, что события, опубликованные во внутреннюю шину, не попадают в поведения. Это потенциальная проблема, если мы хотим, чтобы поведения реагировали и на внутренние события. Но в текущей логике все внешние события приходят через внешнюю шину и затем публикуются во внутреннюю — так что поведения получают их через роутер, а внутренние подписчики (воркеры) получают их через внутреннюю шину. Но поведения не подписаны на внутреннюю шину, поэтому они не видят события, которые публикуются напрямую во внутреннюю шину (например, если какой-то код внутри ячейки вызовет InternalBus.PublishAsync без прохождения через внешнюю шину). Это может быть не ошибкой, а архитектурным решением. Однако, если мы хотим, чтобы поведения обрабатывали все события ячейки, нужно либо подписать их на внутреннюю шину, либо всегда публиковать во внутреннюю шину после роутера (что уже сделано: в ExecuteAsync после роутера вызывается InternalBusField.PublishAsync(@event)). Так что все внешние события всё равно попадают во внутреннюю шину, а значит, подписчики внутренней шины (воркеры) их получают. Поведения же получают их через роутер до публикации во внутреннюю шину. Это нормально.

В текущей архитектуре это избыточно и может мешать, так как события обрабатываются и поведениями, и воркерами одновременно. Это может привести к двойной обработке. Это явная ошибка: воркеры должны быть либо удалены, либо их логика должна быть перенесена в поведения. 🔍 Найденные нюансы
1. Мёртвый код в Worker.cs
Метод OrchestrateWithBehaviorsAsync всё ещё присутствует в файле, но нигде не вызывается.
Это увеличивает объём кода и может ввести в заблуждение.
Рекомендация: удалить этот метод полностью.

2. Хардкод в воркерах (ChannelBackgroundService и TplBackgroundService)
Внутри этих фоновых сервисов есть switch по EventType, который логирует и обрабатывает события.
Но теперь эту логику выполняют поведения (HeartbeatBehavior, DataProcessingBehavior).
В результате каждое событие обрабатывается дважды: один раз в поведении, второй раз – в воркере.
Хотя воркеры только логируют (не меняют состояние), это загрязняет логи и может запутать при отладке.

Рекомендация:

Либо закомментировать всю логику внутри ProcessEventAsync (оставить только _logger.LogInformation без switch).

Либо удалить подписку воркеров на внутреннюю шину (убрав _subscription = _internalBus.Subscribe<FractalEvent>(...)). Тогда воркеры будут просто пустыми BackgroundService, которые ничего не делают – их можно вообще не создавать, если они не нужны.

Если воркеры не используются, можно уменьшить BackgroundServiceCount до 0 (но тогда ячейка не будет иметь фоновых процессов, что допустимо, если всё делают поведения).

3. В FractalCellFactory дважды вызывается AttachToAsync для поведений
В методе CreateWithBehaviorsAsync поведения добавляются через конструктор ячейки (AddBehavior), а затем для каждого поведения вызывается await behavior.AttachToAsync(cell, ct).
Это не ошибка, так как AttachToAsync устанавливает ссылку на ячейку и вызывает OnAttachedAsync.
Но если в будущем в AttachToAsync появится подписка на внутреннюю шину (как мы обсуждали ранее), то поведение получит события дважды – один раз через роутер, второй раз через подписку.
Пока это безопасно, но стоит иметь в виду.

4. В FractalCellTemplate.ExecuteAsync роутер вызывается до публикации во внутреннюю шину
Это правильно: поведения получают событие первыми, а затем оно уходит в воркеры.
Но если какое-то поведение изменит событие или отменит его обработку, воркеры всё равно получат исходное событие.
Если такая логика не требуется, это нормально.

5. Отсутствие обработки ошибок в BackgroundLoopAsync оркестратора
В OrchestratorBehavior есть catch (Exception ex), но он только логирует ошибку и продолжает цикл.
Это хорошо, но если ошибка критическая (например, BroadcastAsync упал), можно добавить задержку перед повторной попыткой, чтобы избежать бесконечного цикла с ошибками.
В текущем коде ошибка не прерывает цикл, но после неё сразу идёт await Task.Delay(_interval, ct) – это уже есть. Так что всё ок.
